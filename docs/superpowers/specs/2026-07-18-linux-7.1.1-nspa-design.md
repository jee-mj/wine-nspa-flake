# Linux 7.1.1 NSPA kernel migration

## Selected baseline: exact RT pairing

Use Linux 7.1.1 with kernel.org's exact `patch-7.1.1-rt2` patch. This is the
smallest deterministic migration because the source and PREEMPT_RT patch have
the same baseline. It does not use a newer Nixpkgs kernel source and does not
forward-port an RT patch.

## Scope

- Update this repository's `flake.nix` to fetch Linux 7.1.1 and apply
  `patch-7.1.1-rt2`.
- Retain the exported `linux-nspaPackages` interface.
- Rebase and semantically validate each ordered NSPA patch against the selected
  Linux 7.1.1 + PREEMPT_RT baseline. The ordered sequence is:
  1. `1001-ntsync-preempt-rt-lock-hardening.patch`
  2. `1002-ntsync-priority-ordered-waiter-queues.patch`
  3. `1003-ntsync-mutex-owner-pi-boost.patch`
  4. `1004-ntsync-channel-thread-token-alloc-hoist.patch`
  5. `1007-ntsync-shared-boost-channel-recv-fixes.patch`
  6. `1009-ntsync-channel-entry-refcount.patch`
  7. `1010-ntsync-aggregate-wait.patch`
  8. `1011-ntsync-channel-try-recv2.patch`
  9. `1012-ntsync-channel-recv-snapshot-pop-fields-uaf-fix.patch`
  10. `1013-ntsync-dedicated-slab-caches.patch`
  11. `1014-ntsync-channel-send-pi-lockless-target-scan.patch`
  12. `1015-ntsync-wait-q-kmem-cache.patch`

## Constraints

Patch application is not proof of compatibility. Each NSPA patch must be
reviewed against the 7.1.1 PREEMPT_RT code it changes. A patch that is already
upstream, conflicts, applies with offsets or fuzz, or changes semantics must be
resolved deliberately and recorded before the package is built.

Forward-porting the PREEMPT_RT patch is out of scope. The NSPA patch series is
intentionally rebased and semantically forward-ported from its existing Linux
7.0.12 baseline to Linux 7.1.1.

## Pre-deployment gates

1. Verify every kernel-source and patch hash.
2. Apply the RT patch and every NSPA patch individually; record offsets, fuzz,
   rejects, and manual rebases.
3. Build the complete kernel derivation.
4. After configuration normalization, compare the final configuration to the
   existing NSPA configuration and explain every material difference. Require
   `CONFIG_PREEMPT_RT=y`, `CONFIG_NTSYNC=y` or `CONFIG_NTSYNC=m`,
   `CONFIG_HIGH_RES_TIMERS=y`, the intended timer frequency, the intended
   `NO_HZ` mode, and all existing NSPA-critical options. Do not accept a build
   where normalization silently drops or changes an essential option.
5. Verify the resulting `version` and `modDirVersion` from built outputs.
6. Build `linux-nspaPackages.kernel` and the reproducible flake check for that
   kernel.
7. Build the downstream NixOS toplevel so NVIDIA is compiled for this exact
   kernel.
8. Build any specifically identified external kernel modules not already pulled
   into the downstream toplevel.
9. Record the current known-good NixOS generation and store path; confirm it
   remains available before activation and do not garbage-collect it until all
   post-reboot checks pass.
10. For initial deployment, activate with `nixos-rebuild boot`, not `switch`.
11. After reboot, validate realtime state, ntsync, NVIDIA, the graphical
   session, PipeWire/JACK, and a real Wine-NSPA workload.

If any gate fails, stop deployment and use the previous bootable NixOS
generation as rollback.

## Implementation boundary

Implementation stops after the package changes, flake checks, kernel and
downstream toplevel builds, and commit. Deployment, activation, reboot, and
post-reboot acceptance testing require a separate explicit instruction.
